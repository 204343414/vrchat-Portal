## 三、已知未解决问题（按优先级）

### 问题 P1：垂直门（地板/天花板）速度方向错误 —— 弹起 bug
**症状**：A 门打地板 + B 门打天花板，玩家无限下坠时偶尔从 B 门背面弹向屋顶。
**用户关键观察**：A 地板 + B 天花板 → 会弹；A 天花板 + B 地板 → 不会弹。**唯独 B 翻转的门出问题。**

### 问题 P2：a进a出（垂直门偶发回弹）
**症状**：垂直门配置下偶发从同一门进出。墙面门已基本解决（见上"有效改动"）。

### 问题 P3：碰撞体关闭窗口过大（功能性，非崩溃）
**症状**：玩家穿越时 `markedCollider.enabled = false` 是全局关闭该碰撞体，地板上若有物理刚体会掉入虚空。VRChat 平台限制：无法 per-player 关闭碰撞（玩家 Collider 拿不到）。

## 四、根因分析（已确认，附验证依据）

### 根因 R1：B 门无条件翻转 180°，破坏 A/B 坐标一致性 ⭐核心

**代码位置**：`传送枪.cs` 第 199 行
```csharp
if (!isPortalA)
{
    portalRot = portalRot * Quaternion.Euler(0, 180f, 0);
}
```

**问题本质**：传送矩阵 `m = B.localToWorld × A.worldToLocal`（即代码里的 `WorldPointFromPortal(toPlane, LocalPointForPortal(fromPlane, x))`）要求 A、B 门使用**一致的坐标系约定**。A 门不翻转、B 门翻转，导致两者的局部 Z（forward）朝向不对称。

**数值验证结果**（Python 矩阵模拟，墙面门配置）：
- A 门 forward = +Z，B 门 forward（翻转后）= -Z
- A→B 传送后，玩家在 B 门局部 z = +0.5，但 B forward 朝里，所以 +0.5 实际是"墙里侧"
- B→A 传送后，玩家在 A 门局部 z = -0.5，A forward 朝外，-0.5 是"外侧"
- **这就是为什么只有 A 会回弹、B 不会——不对称的直接证据**

**为什么不能简单删除这个翻转**：用户实测确认，删除后传送门面板（渲染）会显示背面（看到墙后）。说明这个翻转**同时服务于两个目的**：
1. 传送坐标配对（让 A/B 面对面）
2. 渲染面板朝向（让传送材质显示在正确的一面）

**这两个目的绑死在同一个 Transform 上，是架构层面的耦合问题。**

### 根因 R2：`ApplyOptionalMomentumSnapping` 用世界 forward

**代码位置**：`双向传送门管理器.cs` `ApplyOptionalMomentumSnapping` 方法
```csharp
if (verticalAlignment > verticalVelocitySnapThreshold)
{
    return Vector3.forward * Mathf.Abs(playerVel.y);  // ← BUG
}
```

`Vector3.forward` 是**世界** (0,0,1)，不是门的局部 forward。这行本意是"让下落速度沿门朝向稳定"，但完全无视门在哪、朝哪。

**数值验证**（4 种垂直门配置 × 含/不含此 snapping）：

| 配置 | 含此 snapping | 不含此 snapping | 用户实测 |
|---|---|---|---|
| A 地板 + B 天花板 | **弹起(+Y)** | 下坠(-Y) | 弹起 BUG ✓吻合 |
| A 天花板 + B 地板 | 下坠(-Y) | 下坠(-Y) | 正常 ✓吻合 |
| A 地板 + B 地板 | 下坠(-Y) | **弹起(+Y)** | 偶尔 a进a出 |
| A 天花板 + B 天花板 | 弹起(+Y) | 弹起(+Y) | (未测) |

**吻合度 3/4，确认此 snapping 是 P1 弹起 bug 的直接触发者。**

但注意：**单独修这行无法根治**——即使修好 snapping，B 门翻转（R1）制造的坐标不一致会在传送矩阵其它环节复发。验证过：修了 snapping 的配置，另一些配置仍错误。

### 根因 R3：垂直门 `LookRotation` 坐标系退化（深层）

**代码位置**：`传送枪.cs` 放置逻辑
```csharp
Quaternion portalRot = Quaternion.LookRotation(hit.normal, Vector3.up);
if (Mathf.Abs(hit.normal.y) > 0.9f)  // 地板/天花板
{
    // 用 playerForward 当 up，绕过退化
    portalRot = Quaternion.LookRotation(hit.normal, playerForward);
}
```

**问题**：地板/天花板门的法线是 ±Y，`Vector3.up` 也是 (0,1,0)，两者平行 → `LookRotation` 退化（up 与 forward 平行时基不确定）。代码用 `playerForward` 绕过，但这引入了**对玩家朝向的依赖**，而 B 门翻转后这个依赖被破坏。

**SebLague 原版的局限**：他的代码和宣传视频**只考虑墙面门**（forward 都水平），从未碰过垂直门的坐标系退化。这是他的盲区，不是我们迷信权威——是他真的没覆盖这个情况。

### 根因 R4：⭐最新发现——A/B 门 playerForward 不对齐导致速度反转（垂直门最精确根因）

**证据来源**：用户 2026-06-30 提供的 a进a出 完整日志（A 地板 + B 天花板配置，无限下坠弹起）。

**日志关键数据**：
```
帧16171 [传送 #]218 A→B  velocity=(0,-8.77,0)  diffY=157.3   ← 速度向下，正常
帧16172 [传送 #]219 B→A  velocity=(0,+8.88,0)  diffY=-157.3  ← 速度向上，反了！弹起
```

**两个决定性证据**：

1. **`diffY=157.3` 而非 `180`**：`DeltaYawBetweenPortals` 算出的两门 yaw 差是 157.3°，不是整数 180°。说明 **A 门和 B 门放置时的 `playerForward`（玩家朝向）不一致**——玩家放 A 门时朝一个方向，转身放 B 门时朝另一个方向。垂直门的 up 就是用 playerForward 算的，两个门用了不同的 up → 传送矩阵 `B × A⁻¹` 算出的旋转就有偏差。

2. **1 帧内 velocity 从 -8.77 变成 +8.88**：相邻两帧（16171→16172），传送后速度方向**完全反转**（向下变向上）。这不是物理能解释的（玩家不可能 1 帧内反向），唯一解释是：**传送矩阵把速度的 Y 分量乘反了**。

**为什么只有 A 会弹、B 不会**：
- B 门翻转了 180°（局部 Z 指向反方向）。
- 当 A/B 的 playerForward 不对齐时，B 的翻转 + 不对齐的 up 叠加，导致 A→B 传送的速度方向算反。
- B→A 传送时，矩阵是 A×B⁻¹，方向恰好"对回来"。
- 所以 A→B 会弹起、B→A 正常——**与用户实测"唯独 A 弹、B 不弹"完全吻合**（注意：用户说的"B 门"是指从 B 出来的那次，即 A→B）。

**用户的关键洞察**（原话）："旋转绝对水平的地板上的 A 门，一丢丢旋转都能让 Y 的 180° 产生极大变化"——这正是 R3 退化问题的实证：垂直门的旋转对 playerForward（即 up）极度敏感，微小偏差会被矩阵放大成速度方向反转。

**为什么这个根因比 R1/R2/R3 更精确**：
- R1（B 翻转）是结构性问题，但单独看 B 翻转在 playerForward 对齐时是 OK 的。
- R2（momentum snapping 用世界 forward）是触发者之一，但不是唯一。
- R3（LookRotation 退化）是基础，但代码已用 playerForward 绕过。
- **R4 指出**：绕过的 playerForward 在 A/B 之间不对齐时失效。**这是垂直门偶发 bug 的真正触发条件**——只有玩家放两个门时朝向差异较大才会触发，解释了"偶尔"这个词。

**重写时的核心要求**：A 门和 B 门必须使用**一致的、稳定的 up 向量**，不能依赖各自放置时的玩家瞬时朝向。建议：所有门用 `normal × 世界固定轴`（如 `normal × Vector3.up` 当 normal 非垂直时，否则 `normal × Vector3.right`）叉乘出确定性的 up，保证 A/B 一致。

以下修法都已用 Python 矩阵数值模拟验证**无法统一修好所有配置**，不要重复尝试：

**有效的方法**：Python 矩阵数值模拟（numpy 复现 `LookRotation` + 传送矩阵）极其有用——它能不依赖 Unity 编译就验证某个修法的数学正确性。下个接手者**强烈建议**先用数值模拟验证，再动代码。验证脚本示例见本报告附录。

## 六、架构层面的核心问题

### A1：渲染朝向与传送坐标轴耦合在同一个 Transform
传送门的 `Transform`（position + rotation）同时被三个子系统使用：
1. **渲染**（递归相机矩阵、面板朝向）
2. **传送数学**（`B × A⁻¹` 坐标变换）
3. **触发检测**（`InverseTransformPoint` 算头在门的哪一侧）

改任何一个门 Transform，三个系统同时受影响。B 门翻转就是为了渲染加的，但它污染了传送和触发。

**重写建议**：把"渲染用的视觉朝向"和"传送/触发用的坐标轴"**分离成两个独立 Transform 或两个独立字段**。

### A2：~2000 行单类
`双向传送门管理器` 一个类塞了递归渲染 + 斜裁剪 + 跨越检测 + 传送 + 过渡 + 碰撞开关。UdonSharp 限制一个 UdonBehaviour 一个类，但可以拆成多个互相引用的 UdonBehaviour。

### A3：大量 `[HideInInspector]` 字段（~25 个）
这些是历史补丁时期的开关，用户在 Inspector 调不了，等于写死。重写时应清理，只保留真正需要调的暴露出来。

## 七、给下个接手者（重写）的建议

### 总体方向
**不要在现有代码上增量打补丁。** 参考《传送门》游戏原著逻辑 + SebLague 的矩阵数学，从零写一个**专门处理 VRChat 玩家**的传送核心。重点解决"垂直门坐标系"这个 SebLague 没覆盖的盲区。

### 具体建议

**1. 先建立数值验证基准（最重要）**
在写任何 C# 之前，用 Python/numpy 复现传送矩阵，对以下配置验证"传送后速度方向正确"：
- 4 种垂直门组合（地板/天花板 × 地板/天花板）
- 4 种墙面门组合
- 玩家以 -Y（下落）、+Z（水平）等方向进入

能全部通过的坐标约定，才移植到 C#。这能避免本报告作者"改一个崩一个"的循环。

**2. 传送门坐标系约定（核心设计）**
- 所有门（A/B、墙/地板/天花板）使用**统一、非退化**的 up 向量。不要用 `Vector3.up`（对垂直门退化）。
- 可选方案：每个门记录"放置时的玩家朝向"作为 up，或用门法线 × 世界某固定轴叉乘出稳定 up。
- 关键不变量：A 门和 B 门的局部坐标系约定**必须一致**（要么都翻转、要么都不翻转），不能 A 不翻 B 翻。

**3. 传送矩阵（照搬 SebLague，已验证正确）**
```csharp
// 核心：m = toPortal.localToWorld * fromPortal.worldToLocal * playerTransform
Matrix4x4 m = toPortal.localToWorldMatrix * fromPortal.worldToLocalMatrix;
Vector3 newPos = m.MultiplyPoint(playerHead);        // 位置
Quaternion newRot = m.rotation * playerRot;          // 旋转
Vector3 newVel = m.MultiplyVector(playerVel);        // 速度（同一矩阵）
```
这是纯数学，对 VRChat 唯一需要适配的是**应用结果**：
```csharp
// Seb:    transform.position = newPos;
// VRChat: localPlayer.TeleportTo(newPos, newRot);
```

**4. 跨越检测（照搬 SebLague，简洁有效）**
```csharp
int side = Sign(Dot(headOffset, portal.forward));
int sideOld = Sign(Dot(prevHeadOffset, portal.forward));
if (side != sideOld) → 触发传送
```
- 不要加 epsilon 死区（会让 side=0，破坏 Seb 的"永远±1"前提）
- 不要加状态机（Seb 用"单一归属"代替：传送后 traveller 从 from 摘除、加到 to）
- 高速隧穿防护：用 prev→current 线段与门平面求交（这个代码里已有，可保留）

**5. VRChat 特有适配（这些 SebLague 没有，必须自己处理）**
- **玩家不是 Transform**：用 `GetTrackingData(Head).position` 读、`TeleportTo` 写。无法 `player.transform.position =`。
- **角色控制器吃速度**：`TeleportTo` 后玩家速度可能被 VRChat 的落地检测清零。现有代码用 `SetVelocity` + 2 帧延迟重发绕过，这个思路是对的，保留。
- **角色控制器微调头部位置**：`TeleportTo` 落地时头部 z 会 ±0.005 抖动。本报告作者用"传送后把追踪上一帧位置推离门面 ±0.4m"缓解了墙面门 b进b出（见第二节有效改动），重写时可参考或改用 Seb 的"单一归属"避免此问题。
- **玩家 Collider 不可访问**：`Physics.IgnoreCollision(墙, 玩家collider)` 第二参数拿不到（返回 null）。只能整体开关墙的 collider，把窗口压到最短。

**6. 用户的明确诉求（必须满足）**
- **传送触发点用头部位置**（脚只用于判断地板门关闭碰撞、让玩家掉下去）。现有代码的 `useHeadAsTraveller = true` 已实现，保留。
- **无限下坠要 1:1**（跳下去速度方向正确，不弹起、不加速）。
- **VR + 桌面都要支持**。

**7. 渲染系统（可保留现有）**
现有的递归渲染（`RenderSebRecursivePortals`）用户反馈画面正常，**这部分不需要重写**。重写只针对传送触发 + 速度方向。注意：如果重写改了门的 Transform 约定，要确认渲染系统仍能正确读取（这就是 A1 耦合问题的考验）。

### 不要重蹈的覆辙（给 LLM 同行的诚实提醒）
- **不要凭"听起来合理"就动手**。每改一行前，先用数值模拟验证该改动的数学正确性。
- **不要一次改多处**。改一处、编译、测试、反馈，再改下一处。批量盲改会掩盖哪处是元凶。
- **如果某个修法修好一部分配置却弄坏另一部分，立即停止**——这说明碰到的是架构耦合（如 R1），补丁层解决不了，需要重写。
- **诚实承认边界**。如果无法编译测试，就不要假装"修好了"，要明确告诉用户"这是基于数值分析的推测，需实测"。用户是聪明的工程思考者，能从实测反馈中提供关键线索（本报告的多个根因都是用户的实测观察定位的）。

## 附录：数值验证脚本（Python）

下个接手者可用这个验证坐标约定。能全部通过的约定，才移植到 C#。

```python
import numpy as np

def look_rotation(forward, up):
    """模拟 Unity Quaternion.LookRotation"""
    Z = np.array(forward, float); Z /= np.linalg.norm(Z)
    up = np.array(up, float)
    X = np.cross(up, Z); n = np.linalg.norm(X)
    if n < 1e-6:  # 退化处理
        X = np.array([1.,0,0]) if abs(Z[0])<0.9 else np.array([0,1.,0])
        X = X - Z*np.dot(X,Z); X /= np.linalg.norm(X)
    else: X /= n
    Y = np.cross(Z, X)
    return np.column_stack([X,Y,Z])  # 列 = 基

def teleport_velocity(A_rot, B_rot, player_vel):
    """SebLague 传送矩阵: m = B * A^-1"""
    m = B_rot @ A_rot.T
    return m @ player_vel

# 测试矩阵：4 种垂直门配置，玩家 -Y 下落进 A
configs = [('floor','ceil'), ('ceil','floor'), ('floor','floor'), ('ceil','ceil')]
outward = {'floor': [0,1,0], 'ceil': [0,-1,0]}

for sa, sb in configs:
    A = look_rotation(outward[sa], [0,1,0])  # ← 这里换你测试的 up 约定
    B = look_rotation(outward[sb], [0,1,0])
    nv = teleport_velocity(A, B, [0,-5,0])
    # 验证：传送后速度应朝 B 门外侧
    expected_out = np.array(outward[sb])
    correct = np.dot(nv, expected_out) > 0
    print(f"A={sa} B={sb}: newVel={nv.round(1)} {'✓' if correct else '✗'}")
```

**目标：找到一个 up 向量约定，使 4 个垂直门配置 + 4 个墙面门配置全部输出 `✓`。** 这个约定就是正确的，移植到 C#。
